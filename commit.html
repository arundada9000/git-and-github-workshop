Git revert & reset: undo almost anything

A complete, scenario‑based guide to reverting changes in Git

1. Why reverting matters

Git is a time machine. Mistakes happen: you commit too early, add a bug, or accidentally delete something. Git provides powerful tools to undo changes at every stage – from your working directory to remote branches. This tutorial covers all the methods, their risks, and when to use each.

Before you start: Understand the three Git storage areas – working directory, staging area (index), and repository (commits). Reverting means moving content between them or creating new commits that cancel previous ones.

2. Discard changes in working directory (unstaged)

You modified a file but haven't staged it. To throw away those changes and restore the file to the last committed version (or staged version if any):

CommandEffect
git restore <file>
modern		Discards unstaged changes in the working directory for the given file(s).
git checkout -- <file>
classic		Same as above: revert file to staged or HEAD version.
git restore .Discard all unstaged changes in the current directory and subdirectories.
 
# Example: undo changes to index.html
git restore index.html

# Undo all unstaged modifications
git restore .
Warning: This permanently deletes your local edits. They cannot be recovered (unless you have them in another branch or stash).

3. Unstage files (from staging area)

You used git add but now want to unstage the file(s) – keep the changes but remove them from the next commit.

CommandEffect
git restore --staged <file>Unstage the file (remove it from index). Changes remain in working directory.
git reset <file>Same as above: unstages the file.
git reset (no file)Unstages all files, but keeps working directory untouched.
 
# Unstage a single file
git restore --staged app.js

# Unstage everything
git reset

4. Undo commits (local repository)

You have made one or more commits that haven't been pushed yet. You have two main tools: git reset (rewrites history) and git revert (adds a new undoing commit).

4.1 git reset – move the branch pointer

git reset moves the current branch tip to an earlier commit, optionally modifying the staging area and working directory. Three modes:

ModeHEADStaging areaWorking directoryUse case
--softmovedunchangedunchangedUndo commit but keep changes staged (ready to re‑commit).
--mixed (default)movedreset to new HEADunchangedUndo commit and unstage changes (keep edits in working dir).
--hardmovedreset to new HEADreset to new HEADCompletely discard all changes after that commit.
 
# Undo the last commit, keep changes staged (soft)
git reset --soft HEAD~1

# Undo last commit, unstage changes (mixed – default)
git reset HEAD~1

# Undo last commit AND discard all changes (hard)
git reset --hard HEAD~1
Danger: git reset --hard permanently deletes uncommitted changes. Also, never reset commits that have been pushed to a shared branch (rewrites public history).

4.2 git revert – safe undo for shared history

git revert creates a new commit that inverses the changes of a previous commit. It does not alter existing history, so it's safe for branches that others use.

# Revert the last commit (creates a new commit)
git revert HEAD

# Revert a specific commit (use its hash)
git revert a1b2c3d

# Revert multiple commits (e.g., last 3)
git revert HEAD~3..HEAD

Git will open an editor to write the revert commit message. The working directory and index are updated to reflect the undo.

5. Undo commits already pushed to remote

If you've pushed a bad commit, you have two options, but one is safer for collaboration.

5.1 Safe: revert on remote

Use git revert locally, then push the new revert commit. This adds a commit that cancels the bad one, and everyone can pull without issues.

git revert HEAD      # revert the latest commit
git push origin main

5.2 Risky: reset and force push

If you are absolutely sure no one else has based work on the bad commit (e.g., your private feature branch), you can reset locally and force push.

git reset --hard HEAD~1   # remove last commit locally
git push --force origin branch-name

Use with extreme caution. Force pushing rewrites history and will cause problems for collaborators.

6. Special case: reverting a merge commit

Reverting a merge is trickier because the merge commit has two parents. A normal git revert of a merge requires you to specify which parent line to keep (mainline).

# Revert a merge commit (hash M)
git revert -m 1 M

-m 1 tells Git to keep the changes from the first parent (the branch you were on when you merged). This reverts the entire merge, but later you may not be able to re‑merge the branch easily. Advanced topic – for most beginners, avoid reverting merges unless necessary, and consult the team.

7. Temporarily go back to an old commit

To inspect or test an older commit without changing branches, you can checkout a commit directly (detached HEAD).

git checkout a1b2c3d

You are now in 'detached HEAD' state. To start a new branch from this point: git switch -c new-branch. To return to your branch: git switch main.

8. Undo a reset – recovering lost commits with reflog

If you accidentally ran git reset --hard and lost commits, Git's reflog records every movement of HEAD. You can find the lost commit and reset back.

# View the reflog
git reflog

# Find the commit you want (e.g., HEAD@{2})
git reset --hard HEAD@{2}

Reflog is local and expires after 90 days – a safety net for local mistakes.

10. Practice exercises

Exercise 1

You have unstaged changes in style.css that you want to discard completely. What command do you use?

Show solution 
git restore style.css

or git checkout -- style.css

Exercise 2

You added debug.log to staging by mistake. You want to unstage it but keep the file as is. What command?

Show solution 
git restore --staged debug.log

or git reset debug.log

Exercise 3

You just made a commit with message "WIP", but you realize you forgot to include a file. You want to undo the commit, add the file, and re‑commit (keeping the changes staged).

Show solution 
git reset --soft HEAD~1
git add missing-file
git commit -m "WIP (fixed)"

Exercise 4

You pushed a buggy commit to the shared main branch. How do you safely undo it?

Show solution 
git revert HEAD
git push origin main

This creates a new commit that reverses the bug, and everyone can pull without history conflict.

Exercise 5

You accidentally ran git reset --hard HEAD~2 and lost two commits. How can you recover them?

Show solution 
git reflog
# find the commit hash before the reset
git reset --hard <hash>

Exercise 6 (advanced)

You merged a feature branch into main, then discovered a bug. You need to undo the merge on the shared main branch. What command?

Show solution 
git revert -m 1 <merge-commit-hash>
git push origin main

This reverts the merge, keeping the first parent (main) as the mainline.

11. Reverting cheatsheet

# Working directory (unstaged)
git restore file
git checkout -- file

# Staging area (unstage)
git restore --staged file
git reset file

# Undo last commit (local, keep changes staged)
git reset --soft HEAD~1

# Undo last commit (local, unstage changes)
git reset HEAD~1

# Undo last commit and discard everything
git reset --hard HEAD~1

# Undo a commit (safe, even if pushed)
git revert HEAD

# Undo a merge commit
git revert -m 1 merge_commit_hash

# Recover lost commits
git reflog
git reset --hard HEAD@{n}

12. Pro tips

  • Always commit your work before trying risky reset operations.
  • Use git status and git log frequently to confirm where you are.
  • Prefer git revert for any commit that has been shared.
  • git reflog is your safety net – it records every HEAD change for 90 days.
  • If you need to discard all untracked files, use git clean -fd (careful – it deletes untracked files).
Git undo mastery – you now have the tools to confidently fix mistakes at any stage.
