Git Internals: under the hood

.git folder, objects, trees, commits – how Git really works

1. The .git directory structure

When you run git init, Git creates a hidden .git folder. This is the core of your repository.

.git/
├── HEAD                 # pointer to current branch (ref: refs/heads/main)
├── config               # repo‑specific configuration
├── description          # used by GitWeb only
├── hooks/               # client‑ or server‑side scripts
├── index                # staging area (binary file)
├── objects/             # all data (blobs, trees, commits)
│   ├── 01/...           # subfolders named by first two SHA‑1 chars
│   └── info/ pack/
├── refs/                # pointers to commits (branches, tags)
│   ├── heads/           # local branches (e.g., main)
│   ├── tags/            # tags
│   └── remotes/         # remote‑tracking branches
└── logs/                # reflogs (history of where HEAD/branches moved)

2. Git object model

Git is fundamentally a content‑addressable filesystem. Every object is identified by a SHA‑1 hash (40 hex characters). There are four types:

Object typeStoresExample
blobfile content (binary or text) – no metadatagit hash-object -w file.txt
treedirectory listing: filenames, modes, and references to blobs/treesgit write-tree
commitmetadata: author, committer, message, parent commit(s), and tree hashgit commit-tree
tagannotated tag: object pointing to a commit with a messagegit tag -a

3. How a commit is formed

  1. git add creates blob objects for each file and updates the index (staging area).
  2. git commit creates a tree object from the index (snapshot of the project).
  3. Then creates a commit object pointing to that tree and the previous commit(s).
  4. Finally updates the current branch reference to point to the new commit.
# Low‑level commands to see the plumbing
git cat-file -p HEAD          # show commit content
git ls-tree HEAD              # show top‑level tree
git rev-parse HEAD            # show commit hash

4. The staging area (index)

The index is a binary file (.git/index) that holds a sorted list of file paths, permissions, and the SHA‑1 of the blob object. It represents what will go into the next commit.

5. Refs and HEAD

A ref is a file inside .git/refs containing a commit hash. Branches are refs under refs/heads/. HEAD is usually a symbolic ref pointing to a branch (e.g., ref: refs/heads/main).

6. Packfiles and garbage collection

To save space, Git packs multiple objects into a packfile (compressed delta). git gc runs automatically to optimize.

7. Practice questions

1. What command would you use to see the type of an object with hash a1b2c3d?

Answer
git cat-file -t a1b2c3d

2. Where does Git store the list of files that are currently staged?

AnswerIn the index file (.git/index).

3. A commit object contains references to what other object(s)?

AnswerA tree object (the root directory) and zero or more parent commit objects.

4. What is the purpose of the refs/heads/ directory?

AnswerIt holds one file per local branch, containing the commit hash at the tip of that branch.
