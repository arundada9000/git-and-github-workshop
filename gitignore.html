.gitignore : the complete guide

Everything you need to learn how to ignore files in Git

1. What is .gitignore and why do we need it?

When you work with Git, many files should never be committed: temporary files, logs, dependency folders (like node_modules), compiled code, local configuration files, and operating system junk (.DS_Store). The .gitignore file tells Git which files or folders to intentionally ignore. It keeps the repository clean, prevents accidental commits of secrets, and avoids clutter for other developers.

Key point: .gitignore only prevents untracked files from being added. If a file is already tracked, adding it to .gitignore has no effect until you untrack it.

2. Where can .gitignore files be placed?

There are several places where ignore rules can live, each with a different scope:

  • Repository .gitignore – a file named .gitignore in the root (or any subdirectory). It is committed and shared with everyone.
  • .git/info/exclude – a file inside the .git folder. It is local to your repository and not committed.
  • Global gitignore – configured with git config --global core.excludesfile ~/.gitignore_global. Applies to all repositories on your machine.

Patterns are cumulative: Git combines all applicable ignore files. A pattern in a child directory overrides parent patterns for that subtree.

3. .gitignore pattern language (full reference)

Each line is a pattern. Blank lines and lines starting with # are ignored. Patterns are read from top to bottom; later lines can override earlier ones with negation.

PatternMeaningExample
filenameIgnores any file or folder named filename anywheresecrets.txt
path/to/fileIgnores the file relative to the location of the .gitignoredocs/notes.txt
folder/ trailing slashIgnores a directory (and its contents) with that namebuild/
*Matches any string, except slash (any file or folder name)*.log
?Matches exactly one character (except slash)file?.tmp
[abc]Matches one character from the set (e.g., a, b, or c)report[0-9].txt
**Matches any number of directories (including zero)**/temp or a/**/b
!patternNegation – re-include a previously ignored file!important.log
\# or \!Literal # or ! at the beginning (escape)\#readme.md

tip Patterns with a leading slash like /todo.txt match only in the same directory as the .gitignore file. Without a slash, they match in any subdirectory.

5. How to check what is tracked, ignored, or untracked

List all tracked files:

git ls-files

See ignored files (existing on disk):

git status --ignored

List only ignored files:

git ls-files --others --ignored --exclude-standard

Check why a specific file is ignored:

git check-ignore -v path/to/file

Test if a file is tracked: git ls-files --error-unmatch file (returns error if not tracked) or simply git status file.

6. Stop tracking a file (without deleting it)

If a file is already tracked and you want to ignore it in the future:

  1. Add the file to your .gitignore (if not already present).
  2. Remove it from Git’s index (the staging area) while keeping the local copy:
git rm --cached <file>

For directories, add -r (recursive): git rm --cached -r folder/.

3. Commit the change: git commit -m "Stop tracking <file> (now ignored)".

Important: After this, the file will be ignored in future commits. However, anyone who pulls your commit will have their local copy of that file removed if they had it tracked. Communicate with your team!

7. Common mistakes & best practices

Mistakes

  • Adding a file to .gitignore after it's already tracked → it will still be tracked. You must untrack it.
  • Using * to ignore everything and then trying to negate a folder – order matters.
  • Forgetting that patterns like build/ ignore the folder everywhere, not just at root.
  • Ignoring .env but accidentally committing it earlier – secrets stay in history.

Best practices

  • Commit .gitignore early, before generating many untracked files.
  • Use global gitignore for OS/editor files (.DS_Store, *.swp).
  • Keep patterns simple; test with git check-ignore.
  • Comment complex rules with #.
  • Use ** sparingly – it can be slow on large repos.

8. Practice exercises

Try to write .gitignore patterns for each scenario.

Exercise 1

Ignore all .log files everywhere, but keep a file named important.log located in the root directory.

Show solution 
*.log
!important.log

The negation rule must come after the broad ignore.

Exercise 2

Ignore any directory named temp (and all its contents) anywhere in the repository. Also ignore debug.log only in the root folder.

Show solution 
**/temp/
/debug.log

**/temp/ matches temp folders in any directory. /debug.log anchors the pattern to the root.

Exercise 3

Ignore all .txt files inside docs/ and its subfolders, but keep docs/readme.txt.

Show solution 
docs/**/*.txt
!docs/readme.txt

Exercise 4

A folder build/ is generated. Ignore everything inside it, except build/version.txt (which you want to track).

Show solution 
build/*
!build/version.txt

build/* ignores all immediate children, but not the folder itself. Then we re-include the specific file.

Exercise 5 (untracking)

You accidentally committed config.local and now want Git to stop tracking it but keep the file on disk. Write the commands.

Show solution 
echo "config.local" >> .gitignore
git rm --cached config.local
git commit -m "Stop tracking config.local"
